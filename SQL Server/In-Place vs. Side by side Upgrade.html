In-Place vs. Side by side Upgrade
In-PlaceSide by side Upgrade
Upgrade Effort
  • Minimum.
  • Upgrade is automated.
  • No data migration is needed.
  • Efforts need to be planned.
  • Manual migration.
  • Data migration required.
Downtime Requires downtime. Minimum or no downtime. Production will be online while migration is taking place.
Configuration Changes Automated; application configuration is not needed since it would use the same connection information.
  • All configuration needs to be migrated over to new SQL Server instance..
  • Applications needs to change configuration to connect to new SQL Server instance.
  • Requires manual check of all configuration (system and user).
Data Transfer
  • No data transfer required.
  • All server integration changes remains intact and gets upgraded by the system (Replication, Linked Server, SQL Server Agent Jobs, etc).
  • Data transfer required and can be planned in staging manner.
  • All server integrations need to be created over to the new SQL Server instance.
Server name and IP No changes in server name and SQL Server instance. All application needs to change their connection to connect to the new server with new IP address.
Testing Will be relatively faster as same testing script and be used once upgrade is completed.
  • Thourough testing required as there are chances to a configuration mismatch.
  • Testing needs to be planned and new testing script required since server instance changed for testing script.
Rollback
  • Usually is rebuilding the whole system.
  • Requires backup of all configuration, databases including system databases.
 Very easy as production system remains online during migration. If testing fails, it can be remediated with no downtime rquired and all application can connect to previous online production system.
